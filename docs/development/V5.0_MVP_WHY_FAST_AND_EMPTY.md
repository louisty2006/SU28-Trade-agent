# 為什麼 v5.0 跑得很快、沒有數據、也沒有真的用 LLM？

你看到的現象是對的：**目前 v5.0 是「框架 MVP」**，故意用假資料、不接真實 LLM，所以會很快、沒有真實分析結果。下面用程式位置說明原因。

---

## 1. 為什麼「那麼快」？

因為：

- **沒有拉真實行情**：沒有呼叫 yfinance / API。
- **沒有呼叫任何 LLM**：沒有打 DeepSeek / OpenAI 等 API。
- **分析模組都是簡化/假資料**：只做簡單計算或回傳固定結構。

所以整條流程幾乎只做記憶體裡的運算，幾秒內就跑完。

---

## 2. 為什麼「沒有任何數據」？

### 2.1 市場數據是假的

**檔案**：`main_v5.py`  
**函數**：`_fetch_and_validate_data()`

```python
def _fetch_and_validate_data(self):
    """获取并验证市场数据"""
    # MVP: 返回模拟数据
    return {
        'AAPL': None,
        'GOOGL': None,
        'MSFT': None
    }
```

這裡**沒有**用 yfinance 或任何 API，直接回傳三個 `None`。  
所以後面所有模組拿到的「市場數據」都是空的。

### 2.2 圖表型態沒有真實 K 線

**檔案**：`analysis/pattern_recognition.py`  
**函數**：`scan_all()`、`detect_breakout()`

- `market_data.get(ticker)` 拿到的是上面傳下來的 `None`。
- `detect_breakout(ticker, None)` 裡有：`if data is None or len(data) < 20: return None`  
→ 所以**永遠不會偵測到任何型態**，候選列表為空。

結果：圖表型態、因果、情緒、Multi-Agent 等，要麼吃假資料，要麼回傳「無」的摘要。

---

## 3. 為什麼「沒有分析今天股票」？

因為：

1. **輸入沒有今日行情**：上面已經說明 `_fetch_and_validate_data()` 只回傳 `None`。
2. **型態掃描沒有候選**：如上，`pattern_recognition` 在資料為 `None` 時不產生候選。
3. **其他分析模組**（因果、情緒、Multi-Agent）在 MVP 裡多是回傳固定摘要或空結果，沒有接真實行情或新聞。

所以整條 pipeline 從頭就沒有「今天真實股票數據」，自然不會有「分析今天股票」的實質內容。

---

## 4. 為什麼「沒有進行所有 LLM」？

### 4.1 決策有走「防幻覺」模組，但沒有真實 LLM

**檔案**：`main_v5.py`  
建立 `ReishiV5` 時：

```python
self.anti_hallucination = AntiHallucination()  # 沒有傳入 llm_client
```

**檔案**：`core/anti_hallucination.py`  
`query_with_protection()` 裡：

```python
if self.llm is None:
    # 如果没有LLM客户端，返回模拟结果
    return self._mock_response(prompt, provided_data)

# 只有 llm 不為 None 才會執行下面這行
raw_response = self._call_llm(protected_prompt, system_prompt)
```

也就是說：

- **沒有任何地方真的呼叫 `_call_llm()`**（沒有打 API）。
- 每次都是 `_mock_response()`：回傳固定內容，例如 `content="模拟分析结果"`。

所以「所有 LLM」在目前版本裡**完全沒有被執行**，只有模擬回應。

### 4.2 `_call_llm` 本身也是佔位

**檔案**：`core/anti_hallucination.py`  
`_call_llm()`：

```python
def _call_llm(self, prompt: str, system_prompt: Optional[str] = None) -> str:
    # TODO: 集成真实LLM API
    # 目前返回占位符
    return json.dumps({
        "facts": [],
        "inferences": [],
        ...
    }, ensure_ascii=False)
```

即便未來傳入 `llm_client`，目前這個函數也還沒接上真實 API，仍是佔位。

---

## 5. 報告裡「⚠️ LLM響應格式錯誤」從哪來？

決策引擎期望的是「決策用 JSON」：`actions`, `hold_positions`, `overall_assessment` 等。

**檔案**：`core/decision_engine.py`  
`decide()` 裡：

```python
response = self.anti_hallucination.query_with_self_critique(...)
decision_data = self._parse_decision_from_response(response.final_analysis.content)
```

而 `query_with_self_critique` 在沒有 LLM 時，整條鏈都回傳的是「防幻覺格式」的模擬結果（例如 `content="模拟分析结果"` 或一小段 JSON，但**沒有** `actions` 等決策欄位）。

**檔案**：`core/decision_engine.py`  
`_parse_decision_from_response()`：

```python
try:
    data = json.loads(response)   # 若 content 是 "模拟分析结果" → 解析失敗
    return data
except json.JSONDecodeError:
    return {
        "actions": [],
        "overall_assessment": "無法解析LLM響應",
        "risk_warnings": ["LLM響應格式錯誤"]   # ← 你看到的警告
    }
```

所以：

- **沒有 LLM** → 回傳的是模擬/防幻覺格式，不是決策 JSON。
- 解析失敗 → 走 `except` → 得到「今日無操作建議」+「⚠️ LLM響應格式錯誤」。

---

## 6. 總結對照表

| 你的觀察               | 原因（程式層面） |
|------------------------|------------------|
| 跑得很快               | 沒拉真實行情、沒呼叫任何 LLM，只跑假資料與佔位邏輯。 |
| 沒有任何數據           | `_fetch_and_validate_data()` 只回傳 `{ticker: None}`，沒有 yfinance/API。 |
| 沒有分析今天股票       | 市場數據是假的 → 型態掃描無候選 → 其他分析也無真實輸入。 |
| 沒有進行所有 LLM       | `AntiHallucination(llm=None)` → 一律 `_mock_response()`，從未呼叫 `_call_llm()`。 |
| 報告出現 LLM 格式錯誤  | 決策引擎解析的是「防幻覺模擬內容」，不是決策 JSON → `JSONDecodeError` → 寫入「LLM響應格式錯誤」。 |

---

## 7. 若要「有數據 + 有 LLM」需要做什麼？

1. **接上真實市場數據**  
   在 `main_v5.py` 的 `_fetch_and_validate_data()` 裡用 yfinance 或現有 `utils/data_fetcher.py` 拉取 OHLCV，回傳真正的 DataFrame（或你專案裡約定好的結構），而不是 `None`。

2. **接上真實 LLM**  
   - 在建立 `AntiHallucination` 時傳入真正的 `llm_client`（例如 DeepSeek / OpenAI 的 wrapper）。  
   - 在 `_call_llm()` 裡實作：發送 `prompt`（與必要時的 `system_prompt`）到該 API，並回傳回覆字串。

3. **決策格式要一致**  
   - 要麼讓 LLM 在防幻覺流程的最終輸出裡，直接回傳「決策用 JSON」（含 `actions`, `hold_positions` 等）；  
   - 要麼在決策引擎裡，從 `ProtectedResponse` / `CritiquedResponse` 的某個欄位（例如 `content` 或專用欄位）再解析出決策 JSON，並確保格式符合 `_parse_decision_from_response()` 的預期，這樣就不會再觸發「LLM響應格式錯誤」。

這樣一來，v5.0 就會變慢（有網路 I/O）、會有真實數據、會真的跑「所有 LLM」並產生意義上的「分析今天股票」結果。

---

**結論**：目前 v5.0 是「架構 MVP」——流程都在，但資料與 LLM 都是佔位，所以跑得快、沒有數據、沒有真正 LLM，報告裡的「LLM響應格式錯誤」來自決策引擎解析了非決策格式的模擬內容。要改成完整版，就需要接真實數據與真實 LLM，並對齊決策 JSON 格式。
